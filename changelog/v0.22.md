> 发布日期：2026-06-11　|　分支：**dev**　|　负责人：henry + AI

技术总监 + 重度用户 + 竞品三视角审核后，先解决会在生产真实出问题的 P0 缺陷。

`ruff` 干净，前端 `build` 通过。

## 一、本次变更

### 🔴 架构（高峰会咬人的）
- **S1 数据集运行改异步**：`/api/datasets/{id}/run/` 不再同步阻塞 gunicorn worker（原先最长 60s+ 会把
  3 个 worker 占满导致整站卡死、客户端 30s 超时）。
  - 后端 `services.start_dataset_run()` 立即返回"运行中"的 DataFile，实际跑数在后台线程执行
    （线程内独立持有并在结束时关闭 DB 连接）。
  - 调度进程仍用同步 `run_dataset()`（独立进程，阻塞无碍）。
  - 前端 `DatasetView` 改为轮询文件状态（每 1.5s，最长 10 分钟），按钮保持 loading。
  - 测试内联开关 `DATASET_RUN_INLINE`（默认 False=异步；测试置 True 便于断言）。
- **S2 SQLite 写健壮性**：元数据库写入热点加 "database is locked" 指数退避重试（`_retry_locked`）；
  WAL `busy_timeout` 提到 20s 与 settings 一致。规模上来仍建议切 MySQL（`DATABASE_URL`，手册 §8.1）。

### 🟠 数据正确性
- **M1 保留只算成功文件**：`_apply_retention` 仅对 `SUCCESS` 计数与按天清理，失败/运行中不再占名额、
  不会因几次失败把好文件挤掉删除。
- **M2 僵尸"运行中"清道夫**：`reap_stuck_running()` 把超过 `STUCK_RUNNING_SECONDS`（默认 30 分钟）
  仍 RUNNING 的文件标记失败（进程崩溃/重启会留下僵尸）；调度进程每 30s reconcile 时顺带清理。
- **M3 权限祖先查询去 N+1**：`Folder.ancestor_ids()` 改为一次性载入 (id,parent_id) 在内存上溯，
  含环保护，消除逐级懒加载的 N 次查询。

### 🔒 安全
- **SEC1 登录限流**：`/api/auth/token/` 加作用域限流（默认 `10/min`，`LOGIN_THROTTLE_RATE` 可调），防爆破。
- **SEC2 Token 过期 + 登出失效 + 登录轮换**：
  - 新增 `ExpiringTokenAuthentication`，Token 绝对有效期 `TOKEN_TTL_SECONDS`（默认 7 天）过期即失效并删除。
  - 新增 `/api/auth/logout/` 删除服务端 Token；前端 `logout()` 先调它再清本地。
  - 登录轮换：每次登录删旧建新 Token，刷新有效期（单活动会话）。
- **M4 Excel 公式注入防护**：导出时对以 `= + - @ \t \r` 开头的字符串单元格加前导单引号，
  防止业务数据被 Excel/WPS 当公式执行（CSV/Excel injection）。
- **M5 SQL 只读校验补盲**：去注释新增 MySQL `#` 行注释；剥离字符串字面量新增双引号串与反斜杠转义引号，
  减少误杀同时保持"宁可误拒"的安全姿态（仍以业务库只读账号为第一防线）。
