# data-nexus 发布与代码提交流程

> 配合 [部署.md](部署.md) 使用。本文回答两件事：**① 服务器要装什么 ② 后续怎么提交代码并更新到服务器**。

---

## 0. 全局流转

```
   你本机(开发)                代码库                      服务器(生产)
  ┌───────────┐  git push   ┌──────────────┐  git pull   ┌──────────────┐
  │ 改代码     │ ──────────► │ 公司内网GitLab│ ──────────► │ gunicorn+定时 │
  │ commit     │             │ (+ GitHub备份)│             │ 跑起来        │
  └───────────┘             └──────────────┘             └──────────────┘
```
核心：**代码库是中转站**。你不把代码直接丢服务器，而是推到代码库，服务器从代码库拉。
> 内网服务器通常连不上 GitHub，所以**服务器从公司内网 GitLab 拉**；GitHub 作为外网备份。

---

## 1. 服务器需要安装的软件（清单）

服务器只是"拉代码 + 跑两个进程"，**不需要任何额外常驻服务**。

### 必须安装
| 软件 | 用途 | CentOS/RHEL/麒麟 安装 |
|---|---|---|
| **git** | 从代码库拉代码 | `sudo yum install -y git` |
| **Python 3.11+** + pip | 运行后端 | `sudo yum install -y python3 python3-pip` |
| **Node.js 18+** + npm | 构建前端 | `curl -fsSL https://rpm.nodesource.com/setup_20.x \| sudo bash -`，再 `sudo yum install -y nodejs` |

### 不需要安装（澄清）
- ❌ Docker、❌ Nginx、❌ PostgreSQL
- ❌ Oracle 客户端 / Instant Client（用 oracledb thin 纯 Python）
- ❌ MySQL 客户端库（用 PyMySQL 纯 Python）

### 服务器已自带（无需装）
- **sshd**（SSH 服务）——你登录服务器、或本机直传代码用，Linux 默认就有。
- **systemd**——用于托管两个进程、开机自启（见 部署.md 第 5 节）。

### 网络要求（要开通的访问）
1. 服务器 → **公司 GitLab**（拉代码）。
2. 服务器 → **业务 Oracle / MySQL**（只读账号取数）。
3. 用户浏览器 → 服务器 `:8000`（或经你们网关的 HTTPS 域名）。

## 2. 首次部署（一次性，详见 部署.md）

```bash
sudo yum install -y python3 python3-pip git
curl -fsSL https://rpm.nodesource.com/setup_20.x | sudo bash - && sudo yum install -y nodejs

cd /opt && git clone <公司GitLab地址> data-nexus && cd data-nexus
bash deploy/setup.sh                      # 自动建环境/装依赖/构建前端/生成.env/迁移
vi backend/.env                           # 填 ALLOWED_HOSTS、CSRF_TRUSTED_ORIGINS
backend/.venv/bin/python backend/manage.py createsuperuser
bash deploy/start.sh                      # 或配 systemd
```

---

## 3. 后续提交代码 → 更新到服务器（日常循环）

### 第 1 步：本机改完，推到代码库
```bash
git add -A
git commit -m "本次改动说明"
git push                # 同时推 GitHub + 公司库（见下节"一次推两个库"）
```

### 第 2 步：服务器拉最新并重启
```bash
cd /opt/data-nexus
git pull
bash deploy/setup.sh    # 依赖或前端有变化时执行；只改后端 .py 可跳过
bash deploy/start.sh    # 或： sudo systemctl restart datanexus-web datanexus-scheduler
```

> **谁做第 2 步？** 两种模式，按你们分工选其一：
> - **运维负责**：你 push 到公司库后通知运维，运维上服务器 pull + 重启。交付给运维的只有【公司库地址 + 本文档 + 部署.md】。
> - **你自己负责**：你 ssh 上服务器自己 pull + 重启，运维只在最初给环境和权限。

## 4. 配置"一次推两个库"（GitHub + 公司库）

本机执行一次（把两个推送地址都挂到 origin 上）：
```bash
# 第一行是现有 GitHub，必须保留；第二行是公司库
git remote set-url --add --push origin https://github.com/henry558090-afk/DataNexus.git
git remote set-url --add --push origin <公司GitLab地址>
```
以后照常 `git push origin dev`，**一条命令同时进两个库**。

> 鉴权：GitHub 用你现有凭据；公司库需其账号权限 + 凭据（HTTP 用「用户名+访问令牌」，或配 SSH key）。

---

## 5. 注意事项

- **不入库的东西**（已 gitignore，不会被 push，也不会传到服务器）：`.env`、`db.sqlite3`、`backend/media/`、`node_modules`、`frontend/dist`、`.claude/skills/`。
  - 含义：服务器的 `.env`、数据库、生成的文件都是**服务器本地独有**的，`git pull` 不会覆盖它们。✅
- **FERNET_KEY**：服务器首次 `setup.sh` 生成后**单独备份**；它不在代码库里，换服务器/重装要用同一把，否则已存数据源密码解不开。
- **定时进程只能一个实例**：用了 systemd 就别再跑 `start.sh`，避免两个调度器重复跑数。
- **数据库迁移**：拉到带新迁移的版本后，`setup.sh`/`start.sh` 会自动 `migrate`，不用手动。
