# data-nexus 开发规范

> 版本：v1.0　|　日期：2026-06-04　|　本规范为团队共同约定，开发前必读，开发中遵守。
> 本文件是 [技术方案.md](技术方案.md) 的配套约束文件。

## 0. 角色与协作约定

- **技术总监（AI）职责**：全面、客观地提供能解决需求的方案；**不无条件附和**，对不合理处必须基于合理性提出异议；给出方案的同时说明风险与取舍。
- **决策方（你）职责**：关键技术决策拍板。
- **协作方式**：本平台**两人一起开发**。AI 出方案与实现，你审阅与决策；每一步可回溯、可中断、可质疑。

---

## 1. 需求与开发流程（红线）

1. **需求不清，必须先问，严禁猜着开发**。任何功能/需求理解不明确时，先停下确认，再动手。
2. 开发顺序：**先完全满足当前需求 → 再考虑扩展性 → 健壮性 → 简洁性**。不在需求未满足时过度设计。
3. ⚠️ **重要修正：安全不属于"以后再说"的范畴，它是需求本身**。SQL 只读校验、参数绑定变量、结果行数上限、密钥管理，**必须 day 1 落地**，不可后置（见第 5 节）。

---

## 2. 变更与删除约束（红线）

1. **开发新功能不得影响旧功能**。若可能影响，必须先判断：是**优化**（更好）还是**风险**（可能坏事），并向决策方说明后再做。
2. **删除任何代码都必须严谨判断，并明确告知**：删了什么、为什么删、影响范围。
3. **严禁以"代码简洁"为由直接删代码**。所有删除都要告知，且走 Git（可追溯），并在变更说明文档中留痕。不让代码"无声消失"。

---

## 3. 编码规范（一律遵循官方文档）

### 通用
- **命名用英文**（变量、函数、类、文件）；**注释用中文**；文档用中文；Git 提交信息用中文（见第 6 节）。
- **跨平台**：生产部署 Linux、当前本地 Windows 开发。路径一律用 `pathlib` / `os.path`，**禁止硬编码 Windows 路径或反斜杠**；换行、编码统一 UTF-8。

### 后端（Python / Django / DRF）
- 遵循 **PEP 8**、**Django 官方编码风格**、**DRF 官方文档** 的用法。
- 工具强制（不靠自觉）：
  - `ruff`（含 PEP8 检查 + import 排序）
  - `black`（格式化）
  - 提交前本地跑一遍，CI 再校验。
- 类型注解尽量补全；公共函数写中文 docstring。

### 前端（Vue 3 / TypeScript）
- 遵循 **Vue 3 官方风格指南**、**Element Plus 官方文档**、**TypeScript 官方** 用法。
- 工具强制：**ESLint + Prettier + TypeScript strict 模式**。
- 组件用 `<script setup>` + Composition API；状态用 Pinia。

> 规范"按官方文档"不是口号，**以上工具的配置文件会进仓库，作为可执行约束**。

## 4. 测试底线

- **核心安全模块必须有单元测试**：`core/oracle_client.py` 的只读校验、参数绑定、流式 fetch。
- 其余功能按需补测试。
- 理由：第 2 节"新功能不影响旧功能"**必须可验证**，没有测试就只能靠人肉回归，不可靠。

---

## 5. 安全红线（数据平台核心，day 1 落地）

1. **SQL 只读三件套**：① 连接用 Oracle 只读账号兜底 ② 执行前校验仅允许单条 `SELECT`（禁止多语句、禁止 DML/DDL）③ 一切动态值走**绑定变量**。
2. **严禁字符串拼接 SQL**（尤其参数化模板的参数），否则注入。
3. **结果行数上限 + 查询超时 + `fetchmany` 流式**，防生产库被拖垮、防内存 OOM。
4. **密钥与敏感配置走 `.env`，不进 Git**（数据源密码、加密 Key、SECRET_KEY）；提供 `.env.example`。数据源密码加密存储。
5. 下载需登录鉴权，URL 临时签名；操作留审计。

## 6. Git 与版本规范

- **仓库**：GitHub 管理。分支策略：
  - `main`：稳定可发布
  - `dev`：集成开发
  - `feature/*`：单功能开发，完成后合 `dev`
- **提交信息**：Conventional Commits（中文描述），类型前缀英文：
  ```
  feat: 新增 Oracle 数据源连接测试接口
  fix: 修复大结果集导出内存溢出
  docs: 更新当前系统文档
  refactor / test / chore ...
  ```
- **版本号**：SemVer `v主.次.修`（如 v0.1.0）。
- **每次向 GitHub 推送新版本时**：必须在 `系统变更说明文档/` 写本次版本变更 + 下一版本计划（见第 7 节）。

---

## 7. 文档维护（第 10 条落地，含防腐约定）

项目维护两类文档目录：

| 目录 | 作用 | 更新时机 |
|---|---|---|
| `系统变更说明文档/` | 每个版本一份：本次变更了什么、下一个版本要做什么 | **每次推 GitHub 新版本时** |
| `当前系统文档/` | 当前系统架构快照（模块、数据模型、接口、部署） | **每次架构/接口变更时同步更新** |

> ⚠️ **防腐约定**：改了架构/接口**必须同步改 `当前系统文档/`**。文档与代码不同步视为未完成。失真的文档比没有更糟。

`docs/` 目录存放长期方案类文档（技术方案、本规范）。

## 8. 关键决策记录（本项目已定）

| 项 | 决策 |

| 部署 | 生产 Linux；当前本地 Windows 开发（代码跨平台） |
| v1 范围 | **把 SQL 跑成 Excel 提供下载**（核心闭环）；Word 报告/参数化/推送为 v2 |
