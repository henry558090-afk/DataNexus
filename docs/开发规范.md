## 0. 角色与协作约定

- **技术总监（AI）职责**：全面、客观地提供能解决需求的方案；**不无条件附和**，对不合理处必须基于合理性提出异议；给出方案的同时说明风险与取舍。开发时**严格按各技术官方文档**的规范写代码。
- **决策方（你）职责**：关键技术决策拍板。
- **协作方式**：本平台**两人一起开发**。AI 出方案与实现，你审阅与决策；每一步可回溯、可中断、可质疑。

---

## 1. 需求与开发流程（红线）

1. **需求不清，必须先问，严禁猜着开发**。任何功能/需求理解不明确时，先停下确认，再动手。
2. 开发顺序：**先完全满足当前需求 → 再考虑扩展性 → 健壮性 → 简洁性**。不在需求未满足时过度设计。
3. ⚠️ **重要修正：安全不属于"以后再说"的范畴，它是需求本身**。SQL 只读校验、参数绑定变量、结果行数上限、密钥管理，**必须 day 1 落地**，不可后置（见第 6 节）。

---

## 2. 变更与删除约束（红线）

1. **开发新功能不得影响旧功能**。若可能影响，必须先判断：是**优化**（更好）还是**风险**（可能坏事），并向决策方说明后再做。
2. **删除任何代码都必须严谨判断，并明确告知**：删了什么、为什么删、影响范围。
3. **严禁以"代码简洁"为由直接删代码**。所有删除都要告知，且走 Git（可追溯），并在变更说明文档中留痕。不让代码"无声消失"。

---

## 3. 功能验收：三维度审核（红线，每个功能/需求完成后必做）

任何功能或需求**开发完成后，必须依次通过三个维度的审核**，AI 须主动逐条自查并向决策方汇报结论：

- **维度一 · 功能正确性**：是否**完全满足**功能需求？**不留任何 BUG、疑点**？当前实现是否是**最优开发方式**（而非凑合能跑）？
- **维度二 · 质量属性**：功能满足后，评估**健壮性**（异常/边界/失败处理）与**扩展性**（后续演进是否容易）。
- **维度三 · 关联影响**：**全面审核与此功能相关的所有功能**，确认**没有影响到其他功能**（呼应第 2 节）。

> 三个维度全部通过，该功能才算"完成"。任一维度存疑，必须暴露、说明、修正，不得隐藏。

## 4. 编码规范（一律遵循官方文档）

### 通用
- **命名用英文**（变量、函数、类、文件）；**注释用中文**；文档用中文；Git 提交信息用中文（见第 7 节）。
- **跨平台**：生产部署 Linux、当前本地 Windows 开发。路径一律用 `pathlib` / `os.path`，**禁止硬编码 Windows 路径或反斜杠**；编码统一 UTF-8。

### 后端（Python / Django / DRF）
- 严格遵循 **PEP 8**、**Django 官方编码风格**、**DRF 官方文档** 的用法。
- 工具强制（不靠自觉）：`ruff`（PEP8 + import 排序）、`black`（格式化）；提交前本地跑，CI 再校验。
- 公共函数补类型注解 + 中文 docstring。

### 前端（Vue 3 / TypeScript）
- 严格遵循 **Vue 3 官方风格指南**、**Element Plus 官方文档**、**TypeScript 官方** 用法。
- 工具强制：**ESLint + Prettier + TypeScript strict**。
- 组件用 `<script setup>` + Composition API；状态用 Pinia。

> 工具配置文件进仓库，作为可执行约束。

## 5. 测试底线

- **核心安全模块必须有单元测试**：`core/oracle_client.py` 的只读校验、参数绑定、流式 fetch。
- 其余功能按需补测试。
- 理由：第 2 节"新功能不影响旧功能"与第 3 节维度三**必须可验证**，没有测试就只能靠人肉回归，不可靠。

---

## 6. 安全红线（数据平台核心，day 1 落地）

1. **SQL 只读三件套**：① 连接用 Oracle 只读账号兜底 ② 执行前校验仅允许单条 `SELECT`（禁多语句、禁 DML/DDL）③ 一切动态值走**绑定变量**。
2. **严禁字符串拼接 SQL**（尤其参数化模板的参数），否则注入。
3. **结果行数上限 + 查询超时 + `fetchmany` 流式**，防生产库被拖垮、防内存 OOM。
4. **密钥与敏感配置走 `.env`，不进 Git**（数据源密码、加密 Key、SECRET_KEY）；提供 `.env.example`。数据源密码加密存储。
5. 下载需登录鉴权，URL 临时签名；操作留审计。

## 7. Git 与版本规范

- **仓库**：GitHub 管理（https://github.com/henry558090-afk/DataNexus.git）。分支策略：
  - `main`：稳定可发布
  - `dev`：集成开发
  - `feature/*`：单功能开发，完成后合 `dev`
- **提交信息**：Conventional Commits（中文描述），类型前缀英文：
  ```
  feat: 新增 Oracle 数据源连接测试接口
  fix: 修复大结果集导出内存溢出
  docs: 更新当前系统文档
  refactor / test / chore ...
  ```
- **版本号方案（按本项目约定）**：
  - **从 `v0.01` 起，每完成一个版本 +0.01 依次累计**（v0.01 → v0.02 → v0.03 …），**不跨版本**。
  - 重大里程碑进位大版本（如 `v1.00`）。例：核心"SQL→Excel"为第一阶段，"数据分析报告"为第二次大版本更新。
- **每完成一个版本**：① 在 `系统变更说明文档/` 新增 `vX.YZ.md`（本次变更 + 下一版计划）② 同步更新 `当前系统文档/系统架构.md`。**两者缺一不可**。
