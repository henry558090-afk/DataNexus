## 1. 项目概述

data-nexus 是一个**轻量级**的企业内部数据共享平台，核心解决"**取数 → 分发 → 分析**"的自动化闭环：业务人员无需写代码，就能按计划自动拿到 Oracle 数据明细表、收到推送、看到分析报告。

> **轻量原则**：能不引入的中间件就不引入；单机可跑；依赖少、部署简单、好维护。后续真有性能压力再平滑升级（SQLite→PG、APScheduler→Celery）。

### 1.1 核心能力

| 能力 | 说明 |
|---|---|
| **定时取数** | 配置 SQL + 定时规则，平台周期性查询 Oracle |
| **参数化自助查询** | 管理员写带参数 SQL，业务用户只填参数自助取数（核心护城河） |
| **Excel 导出** | 查询结果生成 Excel 明细表，进受控下载中心 |
| **Word 分析报告** | 基于取数结果用 python-docx 生成 Word 报告，同样进下载中心 |
| **受控下载 + 推送** | 文件存平台，推送的是「通知 + 鉴权下载链接」，不外泄原始数据 |
| **权限与审计** | 查询、下载、推送留痕，按角色授权 |

### 1.2 典型用户流

```
数据管理员                    业务用户
    │                          │
    ├─ 配置 Oracle 数据源       ├─ 浏览可用查询任务
    ├─ 编写 SQL 查询任务  ───▶  ├─ 手动触发 / 等定时执行
    ├─ 设置定时规则            ├─ 下载 Excel 明细
    └─ 配置推送渠道            └─ 接收推送 / 查看报告
```

## 2. 技术选型（轻量）

| 层 | 选型 | 理由 |
|---|---|---|
| 后端框架 | **Django 5 + DRF** | 自带 ORM/Admin/认证/权限，快速出后台 |
| 任务调度 | **APScheduler**（django-apscheduler） | 进程内定时，规则可存库、动态增删，**无需 Redis/独立 Worker** |
| 元数据库 | **SQLite** | 单文件零配置，存任务/用户/日志；后续可平滑切 PG |
| 业务数据源 | **Oracle** | 用 `python-oracledb` **thin 模式**，免装 Instant Client |
| 前端框架 | **Vue 3 + TypeScript** | Composition API，类型安全 |
| 前端 UI | **Element Plus** | 后台表格/表单场景最成熟，组件齐全 |
| 状态管理 | **Pinia** | Vue 3 官方推荐 |
| 构建工具 | **Vite** | 快 |
| Excel 生成 | **openpyxl**（pandas 可选） | openpyxl 轻；数据量大或要做分析时再上 pandas |

> **关于 python-oracledb thin 模式**：新版官方驱动支持纯 Python 直连 Oracle，**不需要在服务器装 Oracle 客户端**，这是轻量部署的关键。仅当用到部分高级特性时才需 thick 模式（装客户端）。

                            │ REST API (JWT)
┌───────────────────────────▼─────────────────────────────────┐
│                Django + DRF（单进程 Web 层）                  │
│  accounts │ datasource │ query │ execution │ push │ report   │
│                  + APScheduler（进程内定时）                  │
└──────┬───────────────────────────────────┬──────────────────┘
       │                                   │ python-oracledb (thin)
┌──────▼──────┐                     ┌──────▼──────────────────┐
│  SQLite     │                     │   Oracle 业务库          │
│ (平台元数据) │                     │   （只读取数）            │
└─────────────┘                     └─────────────────────────┘
                                            │
                                    ┌───────▼────────┐
                                    │  推送渠道       │
                                    │ 企微/钉钉/飞书/  │
                                    │ 邮件            │
                                    └────────────────┘
```

**执行链路（一次定时取数）：**
```
APScheduler 到点触发
  → 连接 Oracle 执行 SQL（只读）
  → 用 openpyxl 生成 Excel，存本地 media 目录
  → 写执行记录(execution)
  → 若配置推送 → 调推送渠道发送
  → 若配置报告 → 生成分析报告
```

> 全流程在 Django 单进程内完成。只有一个注意点：APScheduler 要确保**只在一个进程里运行**（见 8.1）。

## 6. 关键技术方案

### 6.1 Oracle 连接（python-oracledb thin）

```python
import oracledb
# thin 模式，无需 Oracle 客户端
dsn = oracledb.makedsn(host, port, service_name=service_name)
conn = oracledb.connect(user=user, password=pwd, dsn=dsn)
```

- **只读约束**：建议用只读账号；SQL 执行前做关键字校验，只允许 `SELECT`，禁止 `INSERT/UPDATE/DELETE/DROP/TRUNCATE/...`。
- 加 `fetchmany` 分批读 + 查询超时，避免大结果集撑爆内存或拖垮源库。
- 连接用完即关；轻量场景按需建连即可，无需常驻连接池。

### 6.2 定时调度（APScheduler）

- 用 `django-apscheduler`，job 持久化到 SQLite，规则支持 **Cron**（每天 8:00）和 **Interval**（每 N 分钟）。
- 创建/修改查询任务时同步增删 APScheduler job，可启用/停用，无需重启。
- **关键约束（见 8.1）**：APScheduler 必须只在单个进程内运行，否则多 worker 会重复执行。

### 6.3 Excel 导出（openpyxl）

- Oracle 取数 → openpyxl 逐行写入 → 存本地 `media/exports/`。
- 加表头样式、列宽自适应；超大数据量（>10 万行）用 `write_only` 流式写或拆分文件。
- 下载走 Django 文件响应 + 临时签名，防越权拉取。

### 6.4 数据推送

统一推送接口，按 channel 适配：

| 渠道 | 实现 |
|---|---|
| 企业微信 | 群机器人 webhook |
| 钉钉 | 自定义机器人 webhook（加签） |
| 飞书 | 自定义机器人 webhook |
| 邮件 | Django `send_mail` + Excel 附件 |

推送模板含：任务名、行数、执行时间、下载链接。**注意**：群机器人发文件有大小限制，超限只发下载链接。

### 6.5 数据分析报告（Word，python-docx）

- 用 **python-docx** 把取数结果套进 Word 模板：标题、统计摘要（汇总/同比环比/TOP N）、明细表格、图表图片。
- **图表**：可用 matplotlib 先出 PNG，再插入 Word；或前端 ECharts 截图。
- Word 模板与平台分离（`templates/report_*.docx`），改版式不用改代码。
- 生成的 .docx 同样进**受控下载中心**（6.8），不直接外发。
- v2（可选后置）：接 LLM 对结果做自然语言摘要，写进报告"分析结论"段落。

### 6.6 安全要点

- 数据源密码加密存储，前端不回显明文。
- SQL 只读校验 + 绑定变量，防注入、防误改 Oracle 源库。
- 任务级权限：用户只能看/下载授权范围内的结果。
- 全量审计：谁、何时、下载/推送了哪个数据，留 IP。
- 下载链接临时签名。

### 6.7 参数化查询模板（自助化核心）

把"代办取数"变成"业务自助"，是平台价值的关键。

- 管理员写带**绑定变量**的 SQL，并为每个参数定义控件：
  ```sql
  SELECT * FROM orders
  WHERE dept = :dept AND biz_date BETWEEN :start AND :end
  ```
- 参数元数据（JSON）描述：参数名、类型（文本/日期/下拉）、是否必填、下拉选项（可写死或来自另一条 SQL）。
- 业务用户进来看到的是**表单**（选部门、选日期），填完即跑，看不到也不用懂 SQL。
- **安全红线**：参数一律走 oracledb 绑定变量传入，**严禁字符串拼接**，否则注入。

### 6.8 受控下载中心

平台不让文件"满天飞"，所有产物（Excel 明细 / Word 报告）集中管理：

- 文件存平台 `media/`，**只能登录鉴权后下载**，URL 带临时签名、防越权。
- 每次下载写审计：谁、何时、下载了哪个文件、IP。
- 推送到企微/钉钉/群的，是**「通知文案 + 回平台的下载链接」**，群里不出现原始数据。
- 敏感任务可设"导出权限分级"：普通用户只能在线看汇总，明细下载需更高权限。

## 8. 部署架构（轻量）

```
Nginx (反代 + 前端静态资源)
  ├─ /api/  → Gunicorn/Uvicorn (Django，1 个进程跑 web + scheduler)
  └─ /      → Vue 打包后的静态文件

数据：
  ├─ SQLite 文件（平台元数据，挂卷持久化）
  └─ media/ 目录（生成的 Excel）

外部：Oracle 业务库（只读连接）

容器化：docker-compose 单文件起 web + nginx 即可（无 Redis、无 Celery）
```

### 8.1 ⚠️ APScheduler 单进程约束（必读）

APScheduler 在进程内跑，如果 Gunicorn 开多 worker，**每个 worker 都会触发定时任务 → 重复执行**。轻量场景两种解法（任选）：

1. **Web 单 worker** + 在该进程内启动 scheduler（最简单，内部工具够用）。
2. **Scheduler 独立进程**：Web 多 worker 但不启 scheduler，另跑一个 `python manage.py runscheduler` 专管定时（稍解耦，推荐稍上规模时用）。

> 这是本方案唯一需要特别注意的工程点，文档单列提醒。

## 9. 开发里程碑

| 阶段 | 目标 | 主要交付 |
|---|---|---|
| **M1 骨架** | 跑通 | Django+DRF + Vue+Element 脚手架、JWT 登录、SQLite、docker-compose |
| **M2 取数+导出** | 核心闭环 | Oracle 数据源管理、SQL 任务、手动执行、Excel 下载 |
| **M3 定时调度** | 自动化 | APScheduler 定时、执行记录 |
| **M4 推送** | 分发 | 企微/钉钉/飞书/邮件推送 |
| **M5 报告** | 增值 | 规则式统计报告 + ECharts |
| **M6 权限审计** | 收尾 | 角色权限、审计日志、安全加固 |

### 版本边界（重要）

- **v1 = M1 + M2 + M3**：核心只有一条线 —— **配 Oracle 数据源 → 写 SQL 任务 → 跑 → 出 Excel → 受控下载**（定时可作为 v1 的加分项）。**这条闭环能用，平台就立得住。**
- **v2 = M5（Word 报告）+ 参数化模板 + 推送增强**：增值功能，v1 稳定后再做。

> 一句话：v1 就干"**把 SQL 跑成 Excel 给人下载**"这一件事，做扎实。

## 11. 复杂度地图：哪里简单、哪里会咬人

> 自建主体不复杂，但有 4 个点容易做错且做错很疼，列在此处作为开发红线。

| 模块 | 难度 | 说明 |
|---|---|---|
| 数据源/任务 CRUD | 🟢 简单 | 标准 DRF 增删改查 |
| Excel 导出 | 🟢 简单 | openpyxl 直写 |
| 定时调度 | 🟢 简单 | APScheduler，注意单进程（8.1） |
| webhook 推送 | 🟢 简单 | 拼 JSON 发 HTTP |
| Word 报告 | 🟡 中等 | python-docx 模板 + 图表插入，调版式略费时 |
| **SQL 只读防护** | 🔴 **易错** | 黑名单易被绕过 → **只读账号 + 单语句 + 绑定变量**三件套兜底 |
| **参数绑定** | 🔴 **易错** | 必须 oracledb 绑定变量，**严禁字符串拼 SQL**，否则注入 |
| **大结果集** | 🔴 **易错** | `fetchall` 几十万行会 OOM → `fetchmany` 流式 + 行数上限 |
| **调度多进程** | 🔴 **易错** | 多 worker 会重复触发 → 单进程或独立 scheduler 进程 |
| 行级/列级权限 | 🟠 按需 | 真要做会显著增大工作量，**M1~M5 先不做，M6 再评估** |

**给开发的一句话**：把红色四项的"正确姿势"在 M1 就固化成公共函数（`core/oracle_client.py` 的只读执行、绑定变量、流式 fetch），后续所有取数都走它，错一次就全局安全。
```
