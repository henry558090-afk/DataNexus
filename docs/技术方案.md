## 1. 项目概述

data-nexus 是一个**轻量级**的企业内部**数据文件门户**。它把「**SQL → Excel**」自动化，并按**部门 / 功能**编目，让不同角色的人**各取所需**：

- **普通用户**：进来像逛一个"数据超市"，按部门、按功能找到自己有权限的**数据文件，预览、下载**即可，**不接触 SQL、不接触数据源**。
- **管理员**：在管理端**写 SQL → 选数据源 → 归类 → 运行/定时 → 产出 Excel**，并管理目录、用户与权限。

> **一句话**：把"找 IT 取数、Excel 满天飞"，变成"**自助到门户里按权限下载受控的数据文件**"。

### 1.1 核心能力

| 能力 | 说明 |
|---|---|
| **SQL → Excel** | 管理员写 SQL，运行后生成 Excel 明细表（核心） |
| **按部门/功能编目** | 数据文件归属「部门 → 分类」两级目录，便于查找与授权 |
| **用户端自助下载** | 用户在门户浏览授权范围内的数据文件，预览 + 下载 |
| **定时 + 手动生成** | 数据集可定时自动产出，也可随时手动运行；保留历史版本 |
| **角色与权限** | 管理角色（超管/辅助管理员）+ 数据可见角色（老板/总监/主管/成员） |
| **受控下载 + 审计** | 文件存平台、鉴权下载、全程留痕；不让明细 Excel 失控外流 |
| **Word 分析报告** | （v2 大版本）基于结果用 python-docx 出 Word 报告 |

## 2. 产品形态与信息架构

**同一套系统，登录后按角色渲染不同导航**——这是解决"页面杂乱、用户≠管理员"的关键。

### 2.1 用户端（普通用户）—— 高级、商务、好交互

目标：**像个产品，不像内部工具**。留白、克制的高级浅色配色、卡片化、流畅过渡。

```
┌──────────────────────────────────────────────┐
│  你好，张三   🔍 搜索数据文件…        [头像]   │
├──────────────────────────────────────────────┤
│  ⭐ 我的收藏 / 🕑 最近下载                      │
│                                              │
│  按部门浏览                                    │
│  ┌─ 财务部 ──────┐ ┌─ 销售部 ──────┐          │
│  │ 月度报表 / 日报 │ │ 业绩 / 回款    │   …      │
│  └────────────────┘ └────────────────┘         │
│                                              │
│  进入「财务部 ▸ 月度报表」后，是数据卡片墙：     │
│  ┌─────────────────────────────┐              │
│  │ 📊 应收账款明细表             │              │
│  │ 更新于 今天 08:00 · 1,240 行  │              │
│  │ [预览]        [下载 Excel] ⭐ │              │
│  └─────────────────────────────┘              │
└──────────────────────────────────────────────┘
```

要点：① 部门 → 分类 → 数据文件 三步可达 ② 数据卡片显示元信息（更新时间/行数）③ 预览前 N 行再下载 ④ 收藏与最近下载 ⑤ 友好空状态。

### 2.2 管理端（超管 / 辅助管理员）—— 高效工作台

密度高、表格表单为主，与用户端调性分开。

```
📂 数据目录（可编辑）
🛠 管理
  ├ 数据集     ★ 写SQL → 选数据源 → 预览 → 归属(部门/分类) → 运行/定时 → 出Excel
  ├ 目录管理   部门 / 分类
  ├ 数据源     Oracle 连接（超管与辅助管理员均可配）
  ├ 用户与权限 角色分配（老板/总监/主管/成员）+ 成员授权
  └ 执行记录 / 审计
```

> 「数据集编辑器」是整个系统的**心脏**——交互流程：**选数据源 → 写 SQL（带"预览前 50 行"）→ 指定归属部门/分类 → 命名 → 点运行生成 Excel / 或设定时**。

## 3. 角色与权限模型

**两套正交的角色**：一套管"能不能管系统"，一套管"能看到哪些数据"。

### 3.1 管理角色（管系统）

| 角色 | 能力 |
|---|---|
| **超级管理员** | 全局最高：配数据源、写 SQL、建数据集、管目录、管权限，**且能管理其他管理员（增删辅助管理员）** |
| **辅助管理员** | 与超管**能力相同**（含**配置数据源**），**唯一区别：不能管理超级管理员、不能动全局最高配置** |
| **普通用户** | 不进管理端；只在用户端浏览/下载授权范围内的数据文件 |

### 3.2 数据可见角色（看数据，按部门树，从高到低）

| 层级 | 可见范围 | 配置 |
|---|---|---|
| **老板** | **全部部门**的全部数据 | 全局身份，可给多人 |
| **总监** | 被指派的**多个部门**下全部数据 | 一人可挂多个部门 |
| **部门主管** | 自己**单个部门**的全部数据 | 一人对应其部门 |
| **成员** | 默认仅**被授权**的数据；可单独开启"看本部门全部" | 授权对象可为**个人或角色** → 到分类/数据集 |

> 老板 / 总监 / 主管本质都是"看若干部门的全部数据"，区别仅在**覆盖部门数量**（全部 / 多个 / 一个）。

### 3.3 可见性判定（读取某数据集 D，属分类 C、部门 Dep）

```
按顺序命中即可见：
1. 用户是 超管 / 辅助管理员 / 老板        → 可见（全部）
2. 在 Dep 的角色为 总监 或 主管            → 可见（Dep 全部）
3. 在 Dep 的角色为 成员：
     - 勾选了"看本部门全部"               → 可见（Dep 全部）
     - 否则：存在授权命中 C 或 D           → 可见
     - 否则                               → 不可见（默认拒绝）
```

> **默认拒绝**：没有任何授权 = 看不到。安全优先。
> 总监跨多个部门 = 多条"部门-角色=总监"的关系记录。

## 4. 技术选型（轻量）

| 层 | 选型 | 理由 |
|---|---|---|
| 后端框架 | **Django 5 + DRF** | 自带 ORM/Admin/认证/权限，快速出后台 |
| 任务调度 | **APScheduler** | 进程内定时，规则存库、动态增删，**无需 Redis/独立 Worker** |
| 元数据库 | **SQLite** | 单文件零配置；后续可平滑切 PG |
| 业务数据源 | **Oracle** | `python-oracledb` **thin 模式**，免装 Instant Client |
| 前端框架 | **Vue 3 + TypeScript** | Composition API，类型安全 |
| 前端 UI | **Element Plus** | 组件齐全；用户端做高级商务化定制 |
| 状态管理 | **Pinia** | Vue 3 官方推荐 |
| 构建工具 | **Vite** | 快 |
| Excel 生成 | **openpyxl** | 流式写，支持大结果集 |
| 认证 | **Django 内置用户 + DRF Token** | 轻量够用 |

## 5. 系统架构

```
┌─────────────────────────────────────────────────────────────┐
│                  前端 Vue 3 + Element Plus                    │
│   用户端：数据目录 / 预览 / 下载   |   管理端：数据集 / 目录 /  │
│                                       数据源 / 用户权限 / 审计 │
└───────────────────────────┬─────────────────────────────────┘
                            │ REST API (Token)
┌───────────────────────────▼─────────────────────────────────┐
│                Django + DRF（单进程 Web 层）                  │
│  accounts │ datasource │ catalog │ dataset │ execution │      │
│           permission │ audit  + APScheduler（进程内定时）     │
└──────┬───────────────────────────────────┬──────────────────┘
       │                                   │ python-oracledb (thin)
┌──────▼──────┐                     ┌──────▼──────────────────┐
│  SQLite     │                     │   Oracle 业务库          │
│ (平台元数据) │                     │   （只读取数）            │
└─────────────┘                     └─────────────────────────┘
       │ media/ 存放生成的 Excel 文件
```

**执行链路（一次数据集运行）：**
```
手动点击 / APScheduler 到点触发
  → 连接 Oracle 执行 SQL（只读 + 绑定变量 + 流式 + 行数上限）
  → openpyxl 生成 Excel，存 media/
  → 写 Execution（含文件路径、行数、时间）
  → 该数据集"最新文件"指向本次；旧文件作为历史保留
```

> 全流程在 Django 单进程内完成。APScheduler 须**只在一个进程里运行**（见 §10.1）。

## 6. 数据模型

```
DataSource 数据源（Oracle 连接，密码加密）
   └─ 1:N ─ Dataset

Department 部门
   └─ 1:N ─ Category 分类
                └─ 1:N ─ Dataset 数据集
                              └─ 1:N ─ Execution 执行/数据文件

User 用户
   ├─ 全局标志：is_super_admin / is_assistant_admin / is_boss
   ├─ 1:N ─ DepartmentMembership（部门, 角色∈{总监,主管,成员}, see_all_in_dept）
   └─ 1:N ─ Grant 授权（subject=个人或角色 → target=分类或数据集）

AuditLog 审计（user, action, object, ip, ts）
```

### 关键表字段

| 表 | 关键字段 |
|---|---|
| `DataSource` | name, host, port, service_name, username, **password(加密)** |
| `Department` | name, order |
| `Category` | name, **department_id**, order |
| `Dataset` | name, description, **category_id**, **datasource_id**, **sql_text**, params(JSON), cron/interval(空=手动), owner_id, is_active |
| `Execution` | **dataset_id**, status(成功/失败), row_count, **file_path**, started_at/ended_at, error_msg, triggered_by, is_latest |
| `DepartmentMembership` | **user_id**, **department_id**, role(总监/主管/成员), see_all_in_dept(bool) |
| `Grant` | subject_type(user/role), subject_id, target_type(category/dataset), target_id |
| `AuditLog` | user_id, action(下载/运行/登录…), object_type/object_id, ip, ts |

> `Dataset` 取代旧设计的「QueryTask」。数据源密码必须 **Fernet 加密存储**，不明文落库。

## 8. 关键技术方案

### 8.1 Oracle 连接（python-oracledb thin）
- thin 模式免装客户端；**只读账号兜底 + sql_guard 校验 + 绑定变量**。
- `fetchmany` 流式 + 查询超时 + 行数上限，防 OOM/拖库。（已在 `core/oracle_client.py` 实现）

### 8.2 定时调度（APScheduler）
- job 持久化到 SQLite，支持 Cron / Interval；建/改数据集时同步增删 job。
- **须单进程运行**（见 §10.1）。

### 8.3 Excel 导出（openpyxl）
- 取数 → 流式写 → 存 `media/exports/`；下载走鉴权 + 临时签名。

### 8.4 受控下载中心 + 下载体验（★ 核心，重点打磨）

下载是平台命脉，按"好交互"标准设计，作为一等公民对待。

**卡片交互**：每个数据文件是一张卡片，卡片上有 `[下载]` 一键下最新；点卡片其他区域进**详情页**（预览 + 历史）。

**详情页结构**：
```
数据集名                         [⭐收藏] [🔔订阅(v2)]
部门 ▸ 分类 · 🟢 更新于 今天 08:00
┌ 最新版本：1,240 行 · 86 KB · 时间   [预览] [⬇ 下载 Excel] ┐
数据预览（前 50 行，干净表格，可筛选列）
历史版本：近 N 次，每条单独 [下载]
```

**下载"做好"清单（实现红线）**：

| 细节 | 做法 |
|---|---|
| 预览先行 | 进详情默认显示前 50 行，先看清再下，杜绝盲下 |
| 一键即下 | 点击 loading + 完成 toast「已下载 xxx.xlsx」 |
| 友好文件名 | `数据集名_日期.xlsx`；中文走 **RFC 5987**（`filename*=UTF-8''…`）防乱码 |
| 大文件不卡 | `FileResponse` 流式 + 返回 `Content-Length`，浏览器原生进度条，页面不冻 |
| 新鲜度信号 | 🟢今天 / ⚪N天前 / 🟠可能过期，一眼知数据新旧 |
| 权限校验 | 下载端点走唯一可见性判定（§3.3），越权 **403** |
| 可重下 | 「我的下载记录」历史下载一键再下 |
| 留痕 | 每次下载写审计（谁/何时/哪个文件/IP） |
| 空/失败态 | 「尚未生成」「最新生成失败，请联系管理员」，不白屏 |

**格式**：起步**只 Excel(.xlsx)**；CSV / 在线复制后续按需再加。
**订阅 / 新版本提醒**：设计预留（数据集出新版时通知订阅者），**v2 随数据推送一起实现**。

### 8.5 安全要点（day 1 落地，已部分实现）
- 数据源密码加密；SQL 只读 + 绑定变量；行数上限 + 超时；下载鉴权 + 审计。
- 详见开发规范第 6 节与「复杂度地图」。

### 8.6 参数化数据集（自助化，v2 增强）
- 数据集可带绑定变量参数，用户端渲染为表单，填参自助取数；参数一律走绑定变量，严禁拼接。

### 8.7 Word 分析报告（v2 大版本）
- python-docx 套模板出 Word（统计摘要 + 明细 + 图表），同样进受控下载中心。

## 10. 部署架构（轻量）

```
Nginx（反代 + 前端静态资源）
  ├─ /api/  → Gunicorn/Uvicorn (Django，1 进程跑 web + scheduler)
  └─ /      → Vue 打包静态文件
数据：SQLite（元数据） + media/（Excel 文件）
外部：Oracle 业务库（只读）
容器化：docker-compose 起 web + nginx（无 Redis、无 Celery）
```

### 10.1 ⚠️ APScheduler 单进程约束（必读）
多 worker 会重复触发定时任务。解法：① Web 单 worker 内启动 scheduler（最简）；② 或 scheduler 独立进程。

### 11.1 大改版工程约定（已确认）

1. **分支**：在 **`dev` 分支**开发本次大改版，每个版本独立可验证；**稳定后再合并 `main`**，保护已发布、能跑的 main。
2. **数据库重置**：当前**无真实数据**，大改版直接**清空 SQLite + 重建迁移**，干净落地新数据模型。
3. **Oracle 只读账号**：自 **v0.08**（测试连接）起需要（host/port/service_name/user/password）；**v0.06、v0.07 的后端地基与前端重构不依赖它，可先行**。

## 12. 复杂度地图：哪里简单、哪里会咬人

| 模块 | 难度 | 说明 |
|---|---|---|
| 数据源/数据集/目录 CRUD | 🟢 简单 | 标准 DRF |
| Excel 导出 | 🟢 简单 | openpyxl（已实现） |
| 定时调度 | 🟢 简单 | APScheduler，注意单进程 |
| **SQL 只读防护** | 🔴 **易错** | 只读账号 + 单语句 + 绑定变量（已实现 sql_guard） |
| **参数绑定** | 🔴 **易错** | 必须绑定变量，严禁拼 SQL |
| **大结果集** | 🔴 **易错** | fetchmany 流式 + 行数上限（已实现） |
| **调度多进程** | 🔴 **易错** | 单进程或独立 scheduler 进程 |
| **权限可见性判定** | 🔴 **易错** | 多角色叠加（老板/总监/主管/成员+授权），须集中成一个判定函数，所有目录/下载接口都走它；默认拒绝 |

**给开发的一句话**：安全四项已在 `core/` 固化；**权限判定**也要做成**唯一的可见性函数**（§3.3），任何"列目录/下载"都调用它，错一次就全局漏权。
